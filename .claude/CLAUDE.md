# Claude Code Configuration for ViFoodLabel

## Project Overview

**ViFoodLabel** is a dataset and baseline research project for Vietnamese food product label Key Information Extraction (KIE). This workspace contains:

- Raw images + Label Studio annotations (in `data/`)
- Preprocessing and baseline training scripts (in `scripts/`)
- Core modules for Task 1–3 benchmarking (in `src/`)
- Complete annotation and evaluation documentation (in `docs/`)
- Planning/progress materials (in `.plan/`, git-ignored, local use only)

## Key Files

| File/Folder | Purpose |
|---|---|
| `docs/` | Public documentation: dataset specs, annotation guidelines, baseline descriptions, benchmark task definitions |
| `scripts/` | Data preprocessing, baseline training (Tier A–B), evaluation, and dataset publishing |
| `src/` | Core modules: OCR, NER, relation extraction (heuristic + proposed model), metrics |
| `.plan/` | (Git-ignored) Local planning, progress tracking, and project phases — not part of public repo |
| `data/` | (Git-ignored) Images, annotations, and processed dataset splits — published separately via HuggingFace Hub |

## Important Notes for Assistant Work

1. **No AI Attribution**: This is an academic research repository. Remove or do not add any markers like "Co-Authored-By Claude", "generated by AI", etc. — use standard Python/Markdown conventions only.
2. **Dataset-Focused Documentation**: Public docs (in `docs/`) contain only dataset specs, annotation guidelines, and baseline descriptions. Planning/progress (in `.plan/`) is strictly local and not committed.
3. **Scaffold Status**: Many `src/` and `scripts/` files contain function signatures + `TODO` markers but not full implementations. These are ready for actual coding.
4. **Target Venue**: Q1/Scopus 2026 dataset paper submission. Code/docs should reflect publication standards.

## File Conventions

- **Docs**: All in English except `-vi` suffix files (e.g., `annotation-vi.md` is Vietnamese).
- **Code**: Standard Python with minimal comments (only document non-obvious logic).
- **Markdown**: Plain Markdown; no HTML or custom formatting unless necessary.

## Commit Message Conventions

Follow [Conventional Commits](https://www.conventionalcommits.org/):

- **Format**: `<type>: <short description>`
- **Types**: `feat`, `fix`, `data`, `refactor`, `docs`, `test`, `chore`
- **Subject line**: Under 72 characters
- **Body** (optional, line 2+): More detailed explanation if needed

**Important**: Do NOT include data statistics (sample counts, token counts, entity distributions, etc.) in commit messages. Only describe *what* changed and *why*.

**Examples**:
- ✓ `feat: add relation_model.py for Tier D baseline`
- ✓ `fix: correct LayoutLMv3 checkpoint loading in ner_engine.py`
- ✗ `data: processed 416 images, achieved 0.87 F1 on SER task`
